research_mvp Technical Report

Multi-Agent Runtime Architecture, Workflow, and State Machine

This report organizes the current research_mvp system into one complete technical picture: it is not a conventional HTTP chat backend, but a long-running local tmux-based multi-agent runtime with structured task state, a supervisor control loop, dual-channel shared thread / inbox communication, and a web runtime board for human observation and intervention.

1. Overall Architecture

The core of research_mvp is not packaging a model as a stateless chat API, but keeping the runtime roles online, observable, recoverable, and collaborative over time. The system is composed of a configuration layer, a runtime control layer, a persistent state layer, and a web presentation layer.

The true system of record is the local tmux runtime. The web layer is only the visualization and interaction surface, while the source of truth lives in the .research-mvp-data/runtime/ file tree.

Configuration Layer

runtime.toml defines the session name, workdir, runtime root, codex command, agent list, submit keys, and timeout parameters.

Runtime Control Layer

runtime_cli.py is responsible for up / down / send / delegate / supervise, state maintenance, stalled-task detection, and closed-loop progression.

Persistent State Layer

All state is written to disk in thread.jsonl, tasks.json, state.json, agents/*.json, and inbox/*.

Web Presentation Layer

/runtime and /api/runtime/* expose the shared thread, tasks, and agent state, and allow the human to send messages.

2. Key Components

Component Responsibility
tmux session Hosts the fixed agent windows and keeps the CLI agents running persistently.
runtime_cli.py The control hub of the system, responsible for lifecycle, messages, tasks, and the control loop.
thread.jsonl The globally visible discussion stream for auditability, visibility, and cross-agent context.
tasks.json The structured task-state table that underpins stalled detection and closed-loop progression.
inbox/<agent> The actionable queue for a target agent that actually triggers execution.
/runtime The runtime console for the human. It does not replace tmux, but fills the observability gap.
Design principle: thread manages visibility, inbox manages execution, tasks manage state, supervisor manages progression, and the web board manages observation and intervention.

3. Runtime Mechanics

The persistent runtime process is a tmux session named research-runtime by default. Each agent runs in a fixed window: leader / researcher / trainer. The CLI up command creates or repairs those windows and launches the CLI agent specified by codex_command inside each window.

After startup, each agent first receives a bootstrap prompt rather than entering free-form conversation immediately. That prompt injects the repo root, workdir, runtime root, thread path, inbox path, CLI command, and governance files so the agent works under a unified protocol.

5. Task State Machine

The current runtime is not a full BPMN engine, but it already has the most important minimal state machine. Explicit task_id updates take priority, with heuristic text inference only as a fallback.

Status Meaning Typical Trigger
assigned The task has been created and assigned to an owner. Human requests, leader delegation, or system workflow tasks.
in_progress The owner has started making progress. A progress update carrying the same task_id.
blocked The owner explicitly reports that input, approval, or an external condition is needed. blocker signal
stalled The runtime determines that the task is stalled after a long period without progress. Timeout exceeded and outside the nudge cooldown.
completed The owner has explicitly completed the task. completion signal with same task_id

6. Stalled Detection Logic

This is what turns the system from something that can merely converse into something that can keep moving. The runtime actively decides when a task is no longer in a normal wait state but has actually stalled and needs nudging or escalation.

The result of this logic is that the system no longer relies on the leader to remember to keep pushing. The runtime itself detects stalls and drives recovery.

7. The Three Agent Roles

These are not homogeneous workers. The system is intentionally role-asymmetric: leader owns flow and closure, researcher owns knowledge assets, and trainer owns execution assets.

leader

  • Orchestrator and convergence owner
  • Reads the human request, breaks it down, delegates work, follows up, and requests reviews
  • Not an infrastructure operator by default; should not touch up / attach / supervise
  • Owns closure: the task is not finished until the final report is produced and completion is explicitly declared

researcher

  • Focused on knowledge and analysis outputs
  • Responsible for plans, matrices, research notes, implementation guidance, and analytical assets
  • Outputs are usually integrable research assets rather than the final deliverable

trainer

  • Focused on execution and evidence-chain outputs
  • Responsible for script execution, training runs, experiment records, logs, and metrics
  • Emphasizes reproducibility and verifiable evidence

8. Communication Contract

Protocol Point Current Rule
shared thread Handles visibility, auditability, and cross-agent context, but does not directly wake a worker.
targeted inbox This is what actually triggers execution. To make an agent act, you must send a targeted inbox message.
Default reporting target Workers report to leader by default, not to all.
When to use all Reserve it for shared milestones, major risks, and globally visible updates.
task_id If a message carries a task_id, subsequent progress / blocker / completion updates must carry the same id.

9. Web Board Positioning

/runtime is not a business-facing front end. It is a runtime console that reads runtime file state directly instead of going through the old project abstraction.

  • Message stream: shows recent shared-thread messages with sender, recipient, timestamp, and task_id
  • Agent cards: show online status, target, pending inbox count, last event, and latest message
  • Task view: shows the task list for the current human request, including owner, status, due, and stall_count
The web board does not replace tmux. Its purpose is to align the human's view across thread, task, and agent state, and enable precise intervention when needed.

10. Strengths, Limitations, and Next Steps

Dimension Content
Current strengths Fully local, friendly to tmux and CLI agents, auditable state, a minimal task state machine, stalled recovery, and a shared state layer used by both web and CLI.
Current limitations Some task-state transitions still rely on compatibility heuristics, leader orchestration is still prompt-driven, workflow automation is still rule-light, there is no formal heartbeat / lease protocol yet, and there is no artifact dependency graph yet.
Recommended evolution Write task_id / parent_task_id / status intent explicitly, add pane-output or heartbeat-level health checks, upgrade advance_workflow() into a phase/state graph, add task_id filtering to /runtime, and complete runtime doctor / repair.
Conclusion: research_mvp is no longer just “a few agents chatting.” It is evolving toward a local multi-agent runtime that can keep moving, recover, and remain auditable. Its real core is not the prompt alone, but the combination of long-running tmux processes + a filesystem state layer + a supervisor control loop + a task_id-driven state machine.
